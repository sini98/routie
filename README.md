# Routie (루티)

> 하루 일정을 더 쉽게

> **v1 기준점 (2026-07-08)**: 현재 상태를 요약한 기준 문서는 [V1_BASELINE.md](V1_BASELINE.md)를 참고하세요. 이 README는 상세 기술 문서이고, `V1_BASELINE.md`는 "지금 정상 동작해야 하는 것" 목록입니다.

Routie는 하루 동안 여러 장소를 방문하는 사용자를 위한 일정 관리 서비스입니다. 기존 지도 서비스는 길찾기에는 강점이 있지만, 여러 장소를 하나의 일정으로 관리하거나 방문 순서를 한눈에 확인하는 경험은 부족합니다.

Routie는

- 여러 장소를 하나의 일정으로 관리하고
- 지도에서 방문 순서를 직관적으로 확인하며
- 길찾기는 네이버지도를 이용하는

서비스입니다.

## 기술 스택

- **Next.js 14** (App Router) + **TypeScript**
- **Tailwind CSS** + shadcn/ui 패턴 프리미티브(Button, Input, Textarea, Label, Sheet)
- **dnd-kit** — Drag & Drop 방문 순서 변경 (Pointer + Touch + Keyboard 센서)
- **Framer Motion** — Bottom Sheet, Card Hover, Marker 선택 애니메이션
- **LocalStorage** 기반 날짜별 데이터 저장
- 네이버 지도 JavaScript API v3, **PWA** 대응

## 홈 화면 로고 (`Header.tsx`)

브랜드 컨셉("복잡한 일정을 쉽게 연결해주는 서비스")을 담아 워드마크를 다듬었습니다. 사용자가 제시한 레퍼런스 목업(진한 남색 워드마크 + 파란 핀 i + 파란 슬로건 + 양 끝에 핀이 달린 연한 파란 연결선)을 따라 구현했습니다. 전체 형태/폰트는 그대로 두고 색상과 장식만 다듬었습니다.

- **핀 아이콘 자체 제작(`PinIcon`)**: `lucide-react`의 `MapPin`은 얇은 outline(선) 스타일이라 작게 쓰면 존재감이 약합니다. 대신 핀 몸통(물방울 모양) 경로와 안쪽 원 경로를 하나의 `<path>`로 합쳐 `fill-rule="evenodd"`로 채우는 커스텀 SVG를 만들었습니다 — 안쪽 원 부분이 도넛처럼 "뚫린" 또렷한 실루엣 핀이 되고, `MapPin`보다 작은 크기에서도 또렷하고 존재감 있게 보입니다. 이 아이콘 하나를 워드마크의 i 점과 연결선 양 끝, 총 세 군데에서 재사용해 같은 핀 모티프로 통일합니다.
- **"Routie"의 i → 위치 핀**: i의 점(dot) 대신 이 핀 아이콘을 얹습니다. 폰트가 그리는 점은 모양을 커스텀할 수 없어서, **점 없는 i**(dotless i, 유니코드 `ı`)로 글자를 그린 뒤 그 자리 위에 핀 아이콘을 절대 위치(`absolute`)로 얹는 방식을 씁니다. `R·o·u·t·e` 나머지 글자는 폰트가 그리는 그대로 전혀 손대지 않았습니다. 워드마크 전체 글자색은 짙은 남색(`text-foreground`)이고, **i의 핀만 브랜드 블루(`text-primary`)**로 남겨 "목적지 포인트"라는 의미를 도드라지게 합니다. 이 장식적인 마크업이 스크린 리더에 이상하게 읽히지 않도록, 시각적 표기 전체를 `aria-hidden`으로 숨기고 별도의 `sr-only` "Routie" 텍스트를 둡니다.
- **슬로건**: "하루 일정을 더 쉽게" 문구는 브랜드 블루(`text-primary`) + `font-semibold`로, 위치/폰트 크기(`text-sm`)는 그대로 유지했습니다.
- **슬로건 아래 연결선**: 원 두 개를 직선 하나로 잇는 형태 위에, 양 끝 원 바로 위에 같은 핀 아이콘을 하나씩 얹어 "출발지 핀 — 동선 — 목적지 핀"을 표현합니다(중간 경유지 점이나 다른 장식은 없습니다). 로고/슬로건보다 절대 눈에 띄면 안 되므로, 핀·선·원 모두 브랜드 메인 컬러(`#4F7FFF`)보다 훨씬 연한 블루 **`#C7D3FF`** 한 가지로만 그립니다(디자인 토큰이 아니라 이 컴포넌트 전용 값이라 하드코딩했습니다 — 여기 말고 다른 곳에 재사용할 색이 아닙니다).
- 시각적 위계는 **로고(진한 남색 + 파란 핀) → 슬로건(파란색) → 연결선(연한 파란색)** 순서로 눈에 띄는 정도가 옅어집니다.
- **워드마크 크기/핀 위치 조정**: "Routie" 글자 크기를 `text-2xl` → `text-4xl`로 키워 브랜드명이 화면에서 가장 먼저 눈에 들어오도록 했습니다(슬로건/연결선 크기는 그대로 둬서, 커진 워드마크와의 크기 대비로 위계가 더 뚜렷해집니다). i 위의 핀 아이콘은 크기를 `h-4 w-4` → `h-5 w-5`로 살짝 키우고, 세로 위치를 `top: -0.66em` → `top: -0.28em`으로 i에 더 가깝게 내렸습니다 — 예전엔 핀이 글자 위에 둥둥 떠 있는 느낌이었는데, 지금은 i와 자연스럽게 이어져 보입니다(글꼴의 줄 높이(line-height)에는 실제 글자보다 위로 여백이 있어서, 눈으로 봤을 때 자연스러운 위치를 찾으려면 픽셀 계산보다 실제 렌더링을 보고 맞추는 게 정확합니다).

## 기본 지역 설정

홈 화면은 **오늘 외출 / 지정 외출 / 최근 작성한 일정** 3가지로 단순하게 유지하고, 지역/위치를 정하는 방법은 두 화면의 성격에 맞게 나눠뒀습니다: 오늘 외출은 "현재 위치 기반으로 빠르게 시작", 지정 외출은 "날짜만 먼저 고르고, 실제 지역/장소 검색은 일정 편집 화면에서". 이 둘은 **서로 다른 상태를 씁니다** — 아래 "두 값을 분리한 이유" 참고.

- **오늘 외출 — 현재 위치로 빠르게 시작** (`useAutoLocate.ts`, `OutingScreen.tsx`, `NaverMap.tsx`의 `currentLocationOnly` prop): 오늘 날짜 일정에 장소가 **하나도 없을 때만** 동작합니다(장소가 있으면 절대 건드리지 않습니다).
  - 위치 권한이 이미 **허용**되어 있으면, 화면 진입 시 조용히 현재 위치를 가져와(`useAutoLocate`가 반환하는 `currentLocation`, 컴포넌트 state일 뿐 localStorage에 저장되지 않음) 지도를 그 위치로 중심 이동합니다. 매번 오늘 화면에 들어올 때마다 새로 조회하므로(앱 최초 1회만 물어보는 것과 다름), 다른 지역으로 이동해도 그날의 시작 위치가 최신 상태를 반영합니다.
  - **2단계 측위** (`useAutoLocate.ts`의 `fetchPositionWithFallback`): 1차로 `enableHighAccuracy: true`(GPS 우선, `timeout: 8초`, `maximumAge: 0`)로 시도하고, 실패하면(권한 거부는 재시도해도 의미가 없으니 제외) `enableHighAccuracy: false`(Wi-Fi/IP 기반 표준 정확도)로 한 번 더 시도합니다. 데스크톱/노트북처럼 GPS 칩이 없는 기기에서는 high accuracy 요청이 오히려 응답이 느리거나 타임아웃/실패로 끝나는 경우가 있어서, 그 경우 서울로 완전히 폴백하기 전에 더 가벼운 측위로 한 번 더 시도해 "위치는 못 찾았지만 아예 엉뚱한 도시로 떨어지는" 상황을 줄입니다. 그래도 몇백 미터 오차가 날 수 있으므로, **현재 위치는 어디까지나 지도의 시작점(힌트)일 뿐** — 자동으로 잡은 좌표를 장소로 저장하는 게 아니라 지도 중심만 옮겨주고, 위치를 성공적으로 가져왔을 때는 지도 위에 "현재 위치를 기준으로 지도를 불러왔어요. 정확한 위치는 지도에서 조정해 주세요." 안내를 띄워, 오차가 있을 수 있다는 걸 매번 알려줍니다.
  - 권한이 아직 **결정되지 않은 상태**(prompt)면 자동으로 팝업을 띄우지 않고, 지도 위에 "현재 위치로 시작하기" 안내 버튼을 보여줍니다. 이 버튼을 눌러야(실제 사용자 클릭이 있어야) 브라우저 권한 팝업이 뜹니다 — 화면에 들어올 때마다 원치 않는 팝업이 뜨는 것을 막기 위해서입니다.
  - 권한이 **거부**되었거나 위치를 가져오지 못하면 **서울(`DEFAULT_COORDS`)**을 씁니다 — 아래 이유로 지정 외출에서 검색해둔 지역(`routie:defaultRegion`)은 여기서 폴백으로도 쓰지 않습니다.
  - 지도가 이미 만들어진 뒤에 위치 정보가 나중에(비동기로) 도착해도 반영되도록, `NaverMap.tsx`에 이 값 변경을 감지해 지도 중심을 다시 옮기는 effect가 별도로 있습니다(장소가 없을 때만 동작).
  - **위치 조회 중에도 지도는 바로 보입니다.** 예전에는 `useAutoLocate`의 `status`가 `"checking"`인 동안 지도를 불투명 오버레이로 완전히 가렸는데, 위치 조회가 최악의 경우 8초 x 2단계(고정밀 실패 후 표준 정밀도 재시도)까지 걸릴 수 있어 그동안 화면이 "멈춘 것처럼" 보이는 문제가 있었습니다. 지금은 지도를 서울 기본값 등으로 즉시 그리고, 그 위에 작은 배지("현재 위치를 확인하는 중...")만 얹어 안내합니다. 실제 위치가 도착하면 지도가 `panTo`로 부드럽게 그 위치까지 이동합니다(순간 이동이 아님). 자세한 최적화 내용은 아래 "## 지도 로딩 체감 속도 최적화" 참고.
  - **최종 위치는 항상 사용자가 지도에서 직접 확정한 좌표입니다.** 현재 위치는 지도 중심만 옮길 뿐 그 자체로 어떤 장소로도 저장되지 않고, 실제 장소를 추가할 때는 여느 때처럼 "+" → 장소 추가 → `LocationPicker`에서 검색/직접 탭으로 정확한 지점을 찍고 "이 위치로 선택"을 눌러야 저장됩니다(`PlaceForm.tsx`의 `handleConfirmLocation`이 그 좌표를 그대로 씁니다 — 행정동 이름으로 되짚어 변환하는 단계가 없어서, 자동으로 잡힌 지역명이 틀리더라도 저장되는 좌표 자체는 항상 정확합니다).
- **지정 외출 — 달력은 날짜만, 지역/장소 검색은 일정 편집 화면에서** (`app/calendar/page.tsx`): 실제 사용 흐름이 "날짜 선택 → 새 외출 만들기 → 일정 편집 → 장소 검색"이라, 달력에서 날짜를 고른 직후에 지역 검색창부터 보여주는 건 이 단계에서 의미가 없고 오히려 혼란을 줬습니다. 지금은 아직 일정이 없는 날짜를 고르면 요약 카드에 "예정된 외출이 없어요" + **"+ 새 외출 만들기"** 버튼만 뜨고, 누르면 바로 그 날짜의 외출 화면(`/outing/{날짜}`)으로 이동합니다. 실제 지역/장소 검색은 그 화면에서 "+" → 장소 추가 → `LocationPicker`의 검색창(카페/역/상호명까지 지원, `searchPlaces` + `geocodeQuery` fallback)으로 진행합니다.
  - 예전에는 이 자리에 `RegionSearch.tsx`라는 별도 지역 검색창이 있어서, 검색한 지역을 기본 지역(`routie:defaultRegion`)으로 미리 저장해두는 방식이었습니다. 이제 그 컴포넌트는 삭제했습니다(다른 곳에서도 쓰이지 않았습니다) — 장소를 추가할 때 이미 `LocationPicker`에서 원하는 지역/장소를 검색할 수 있어서, 이 단계를 따로 거칠 필요가 없어졌습니다.
  - 더 예전에는 홈 화면에 이 검색창이 있었고, 그보다 더 예전에는 위치 권한을 거부하면 고정된 8개 도시(서울/대구/부산/거제/제주/광주/대전/인천) 중에서만 골라야 했습니다(목록에 없는 지역은 선택 불가). 지역/장소를 정하는 지점이 이렇게 계속 정리/이동해왔습니다.
- **`currentLocation`(오늘 외출)과 `routie:defaultRegion`(지정 외출/최초 실행)을 완전히 분리한 이유**: 예전에는 `useAutoLocate`가 현재 위치를 얻으면 `routie:defaultRegion`에 그대로 저장했습니다. 그런데 이 값은 (지금은 삭제된) 지정 외출 지역 검색 기능과 **같은 localStorage key를 공유**하고 있어서, 예를 들어 지정 외출 달력에서 (테스트든 실제든) "대구"를 한 번 검색해두면, 그 값이 계속 남아있다가 며칠 뒤 "오늘 외출"에 들어갔을 때도 "저장된 기본 지역"으로 다시 읽혀서 지도가 대구로 뜨는 문제가 있었습니다(실제 위치는 서울이고 권한도 허용돼 있었는데도). `useAutoLocate`가 반환하는 `currentLocation`은 이제 이 훅 안의 컴포넌트 state일 뿐 `routie:defaultRegion`을 읽지도 쓰지도 않고(`NaverMap`에도 `currentLocationOnly`라는 별도 prop으로 전달), `routie:defaultRegion`은 최초 실행(`RegionSetup`) 전용으로만 남습니다. 그래서 `routie:defaultRegion`을 앱 시작 시 강제로 지우는 마이그레이션은 넣지 않았습니다 — 이미 들어있는 예전 값(예: 테스트로 검색했던 대구)이 있어도, 이제 오늘 외출은 그 값을 아예 보지 않으므로 더 이상 영향을 주지 않습니다.
- **앱 최초 진입 시 위치 권한 감지** (`RegionSetup.tsx`, 루트 레이아웃에 마운트): 최초 1회, 조용히 위치 권한을 요청해 허용되면 기본 지역을 저장합니다. 거부/미지원이거나 이미 기본 지역이 있으면 아무 화면도 띄우지 않습니다.
- 이 기본 지역은 **일정에 장소가 하나도 없을 때만** 지도/위치 선택 화면의 초기 중심으로 쓰입니다. 장소가 하나라도 있으면 항상 그 장소 좌표가 우선합니다(`src/hooks/useDefaultRegion.ts`).
- **디버깅 로그**: `useAutoLocate.ts`/`NaverMap.tsx`/`useDefaultRegion.ts`/`OutingScreen.tsx`에 `[Routie][Debug]`로 시작하는 로그가 남아 있습니다. 지도 중심이 왜 그렇게 정해졌는지는 `NaverMap`이 찍는 `finalCenter`/`reason` 로그로 바로 확인할 수 있고, `reason`은 `"currentLocation"`(현재 위치 조회 성공) / `"defaultLocation"`(서울 fallback — 조회 실패·거부·미지원일 때만) / `"savedRegion"`(지정 외출 등에서 defaultRegion을 쓰는 화면) / `"savedItinerary"`(일정에 저장된 장소 기준) 중 하나로 통일돼 있습니다.

### 알아둘 버그 수정: useLocalStorage 인스턴스 간 동기화

한때 "위치 권한을 허용했고 실제 위치는 서울인데, 오늘 외출 화면은 예전에 검색했던 다른 지역(예: 대구)에 멈춰 있는" 문제가 있었습니다. 원인은 하드코딩된 좌표가 아니라 **`useLocalStorage`가 컴포넌트 인스턴스마다 완전히 독립된 React state였다는 것**이었습니다 — `OutingScreen`의 `useAutoLocate`가 현재 위치를 가져와 `setDefaultRegion(...)`을 호출해도, `NaverMap`이 자기 자신의 `useDefaultRegion()` 호출로 들고 있던 예전 값(예: 지역 검색 테스트로 저장돼 있던 대구 좌표)은 그 사실을 전혀 알지 못했습니다. 같은 localStorage key를 쓰는 두 개의 서로 다른 `useState`였을 뿐, 서로 구독하지 않았기 때문입니다.

지금은 `src/hooks/useLocalStorage.ts`에 모듈 전역 pub/sub을 추가해, 같은 key를 쓰는 모든 `useLocalStorage` 인스턴스가 값 변경을 즉시 공유합니다(`useOutings`/`useFavorites`/`useCategories`/`useDefaultRegion` 등 이 훅을 쓰는 모든 곳에 자동 적용됩니다). 어느 한 컴포넌트가 저장하면 같은 key를 구독 중인 다른 모든 컴포넌트가 같은 렌더 사이클 안에서 최신 값을 받습니다.

같은 조사 과정에서 두 번째 원인도 찾았습니다: `useOuting`이 "오늘" 날짜에 처음 들어올 때마다 예시 장소 3곳(`INITIAL_PLACES` — 서울아산병원/잠실 카페/저녁식사)을 자동으로 채워 넣고 있었습니다. 이러면 오늘 일정이 "장소 0개"인 순간이 사실상 없어서, 현재 위치 기준 시작 로직이 거의 항상 건너뛰어지고 — 특히 개발 중 **localStorage를 지워도 매번 예시 데이터가 다시 채워져 현재 위치 동작을 테스트하기 어려웠습니다.** 이 자동 채우기를 완전히 제거해, 이제 오늘 일정은 정말로 비어 있는 채로 시작합니다(장소가 없으면 `EmptyState`가 뜨고, 지도는 `useAutoLocate`가 현재 위치를 기준으로 보여줍니다). **개발 중 테스트 팁**: 브라우저 개발자 도구에서 이 사이트의 로컬 저장소(또는 전체 사이트 데이터)를 지운 뒤 새로고침하면, 예시 데이터 없이 정말 빈 상태로 시작하고 위치 권한이 허용돼 있다면 현재 위치를 기준으로 바로 시작됩니다.

### 추가로 발견/수정한 버그: 삭제했는데 다시 살아나는 문제 (마운트 시점 echo-write 경쟁)

즐겨찾기(또는 일정의 장소)를 삭제해도 새로고침하면 다시 나타나는 버그가 있었습니다. 실제로 겪은 사례: `/favorites`에서 "온실카페"를 삭제해도 `localStorage`에는 계속 남아있었습니다(화면에서는 사라져 정상 삭제된 것처럼 보였는데도).

**원인**: 위 pub/sub 수정 이후에도, `write effect`가 "이 값을 정말 새로 로컬에서 바꾼 건지" 판단하는 `isExternalUpdate` 플래그가 **구독(subscribe)으로 받은 값에만** 걸려 있었습니다. 마운트 시점에 `localStorage`에서 값을 읽어와 `setValue`로 로컬 state에 반영하는 것도 "값이 바뀐 것"으로 잡혀 write effect가 다시 실행되는데, 이때 `isExternalUpdate`가 `false`라 "새로 바뀐 값"으로 착각하고 그 값을 `localStorage`에 그대로 다시 씁니다(무해한 재확인 저장처럼 보이지만). 문제는 `/favorites`처럼 같은 key("routie:favorites")를 쓰는 **여러 `useFavorites()` 인스턴스가 한 화면에 동시에 마운트**되어 있을 때입니다(`FavoritesPage` 자신 + `useCategories()`가 내부적으로 한 번 더 호출하는 `useFavorites()`) — 한 인스턴스가 실제로 삭제하고 저장한 직후, 다른 인스턴스의 마운트 시점 읽기가 (그보다 오래된 값을 들고 있다가) "재확인 저장"을 하면서 방금 삭제된 항목을 **되살려 다시 씀**니다. 이 레이스는 실제 브라우저에서 `console.log` 계측으로 재현·확인했습니다.

**수정**: `src/hooks/useLocalStorage.ts`의 마운트 read effect에서 `localStorage`를 읽어 `setValue`를 호출하기 **직전에 `isExternalUpdate.current = true`로 표시**합니다 — "방금 저장소에서 그대로 읽어온 값이니 다시 쓸 필요 없다"는 뜻으로, 구독으로 받은 값과 동일하게 취급합니다. 이러면 같은 key를 쓰는 인스턴스가 몇 개든, 마운트 시점의 재확인 저장이 다른 인스턴스의 최신 삭제/수정을 덮어쓰는 일이 없습니다. 이 훅을 쓰는 모든 데이터(오늘·지정 외출 장소, 즐겨찾기, 카테고리, 기본 지역)에 공통 적용됩니다.

이 김에 `useOutings.ts`(`normalizePlaces`)와 `useFavorites.ts`(`normalizeFavorites`)에도 방어 로직을 추가했습니다: 저장된 목록을 읽어올 때마다 `id`가 없는 항목엔 새 id를 채우고, 같은 `id`가 중복되면 먼저 나온 것만 남깁니다 — `key`/삭제 필터가 모두 `id` 기준이라, 레거시 데이터에 `id`가 없거나 중복되면 특정 항목만 삭제가 안 되는 것처럼 보일 수 있어 추가한 안전장치입니다.

## 지도 로딩 체감 속도 최적화

오늘/지정 외출 화면 진입 시 지도가 뜨기까지 느리게 느껴진다는 피드백을 반영해 아래 6가지를 손봤습니다. 실제 로딩 시간 자체를 줄이는 것과, "멈춘 것처럼 보이지 않게" 체감 속도를 개선하는 것 둘 다를 목표로 했습니다.

1. **SDK 스크립트 프리로드** (`NaverMapsPreloader.tsx`, 루트 레이아웃에 마운트): 홈 화면 등 앱 어디에 들어오든 네이버 지도 SDK 스크립트(`useNaverMapsLoader`)를 곧바로 로드하기 시작합니다. `useNaverMapsLoader`는 `<script id="naver-map-sdk">` 태그로 중복 로드를 막기 때문에, 실제로 오늘/지정 외출 화면에 들어갔을 때 호출되는 `useNaverMapsLoader`는 이미 로드됐거나 로드 중인 같은 스크립트를 그대로 재사용합니다 — 지도 화면에 들어가는 시점엔 SDK가 이미 준비돼 있을 가능성이 높습니다. 루트 레이아웃(`app/layout.tsx`)에 `<link rel="preconnect" href="https://oapi.map.naver.com" />`도 추가해 DNS/TLS 협상도 미리 끝내둡니다.
2. **지도 API 로딩과 현재 위치 조회는 원래도 동시에 진행됩니다**: `<Map>`은 위치 조회 상태와 무관하게 항상 마운트되고, `useAutoLocate`도 같은 렌더에서 바로 호출되므로 SDK 스크립트 로딩과 GPS/Wi-Fi 위치 조회가 병렬로 진행됩니다. 문제는 이 병렬 진행 자체가 아니라 아래 6번(화면을 가리는 방식)이었습니다.
3. **지도 준비 전 로딩 UI를 스켈레톤으로 교체** (`Map/index.tsx`): SDK 로딩+지도 인스턴스 생성이 끝나기 전(`status === "loading"`)에는 정적인 "지도를 불러오는 중..." 문구만 있던 것을, 은은한 펄스(`animate-pulse`) 그라디언트 배경 + 스피너 + 문구로 바꿨습니다. 가만히 멈춘 화면보다 "뭔가 계속 진행 중"이라는 인상을 줍니다.
4. **불필요한 지도 재생성 없음(확인/유지)**: `NaverMap.tsx`의 지도 생성 effect는 `mapRef.current`가 이미 있으면 항상 즉시 반환합니다 — 화면 재렌더/부모 리렌더가 아무리 일어나도 같은 화면에서는 `naver.maps.Map` 인스턴스를 다시 만들지 않고 기존 인스턴스를 계속 재사용합니다. 새 지도 인스턴스는 그 화면이 실제로 언마운트-재마운트될 때만 생깁니다(예: Bottom Sheet를 닫았다 다시 열기).
5. **마커/폴리라인이 관계없는 리렌더로 다시 그려지던 버그 수정** (`LocationPicker.tsx`): `Map`에 넘기는 `onSelectPlace`/`onPickLocation` 콜백이 매 렌더마다 새로 만들어지는 인라인 함수(`() => {}`, 클로저)였습니다. `NaverMap.tsx`의 마커/폴리라인 effect는 이 콜백들도 의존성 배열에 포함하기 때문에, 예를 들어 위치 선택 화면(큰 지도) 검색창에 한 글자씩 타이핑할 때마다(그때마다 `LocationPicker`가 리렌더되어 콜백 참조가 바뀜) 장소 목록이 전혀 안 바뀌었는데도 마커/폴리라인을 통째로 지웠다가 다시 그리고 있었습니다. `handlePickLocation`/`suggestNameFromCoords`를 `useCallback`으로, `onSelectPlace`는 모듈 스코프 상수 `noop`으로 고정해 참조가 항상 같게 만들었습니다 — 이제 마커/폴리라인은 실제로 장소 목록이나 선택 상태가 바뀔 때만 다시 그려집니다.
6. **"멈춘 것처럼" 보이지 않게 — 위치 조회 중에도 지도를 바로 표시**: `OutingScreen.tsx`/`LocationPicker.tsx` 둘 다 예전에는 `useAutoLocate`의 `status === "checking"`인 동안(최악의 경우 8초 × 2단계) 지도를 불투명 오버레이로 완전히 가렸습니다. 지금은 지도를 즉시 보여주고(서울 기본값 등으로 시작) 그 위에 작지 않은 배경을 가리지 않는 작은 배지("현재 위치를 확인하는 중...")만 얹습니다. 실제 위치가 도착하면 `NaverMap.tsx`가 `setCenter` 대신 `panTo`로 부드럽게 이동시켜, 순간적으로 "튀는" 느낌 없이 자연스러운 전환처럼 보이게 했습니다(지도 최초 생성 시의 초기 중심 지정은 애니메이션이 필요 없어 그대로 `setCenter`를 씁니다).

## 화면 구성

```
/            홈 — 로고/슬로건 + 3개 메뉴(오늘 외출/지정 외출/즐겨찾기) + 최근 작성한 일정 목록
/today       오늘 날짜로 /outing/[오늘 날짜]로 리다이렉트 (진입 시 현재 위치 자동 반영 시도)
/outing/[date]  지도 + 일정 리스트 + 장소 추가 (오늘 외출, 지정 외출 공통 화면)
/calendar    지정 외출 — 월간 달력에서 날짜 선택 → 요약 확인(비어있는 날짜면 지역 검색도 표시) → 일정 화면 이동
/favorites   즐겨찾기 목록 — "오늘 외출에 추가" 버튼 제공
```

`/outing/[date]` 하나의 화면을 "오늘 외출"과 "지정 외출" 양쪽에서 공유합니다. `/today`는 매 요청마다 오늘 날짜를 계산해 `/outing/{YYYY-MM-DD}`로 보내주는 얇은 리다이렉트 라우트입니다.

`/outing/[date]` 화면 상단에는 `< 7월 7일 (월) >` 형태로 이전/다음 날짜 이동 버튼이 있어, 날짜를 하루씩 옮겨가며 각 날짜의 일정을 바로 확인/편집할 수 있습니다(URL도 `/outing/YYYY-MM-DD`로 함께 바뀝니다). 오늘 날짜에는 "오늘" 뱃지가 표시됩니다.

`/outing/[date]`, `/calendar`, `/favorites` 화면 맨 왼쪽 상단의 `<` 버튼은 항상 `router.push("/")`로 **홈으로 이동**합니다(브라우저 히스토리에 `/`를 새로 쌓는 방식이라, 이후 브라우저 자체의 뒤로가기 동작에는 영향을 주지 않습니다).

## 네이버 지도 API 설정 방법

1. 프로젝트 루트에 `.env.local` 파일을 만듭니다 (`.env.local.example` 참고).

   ```bash
   cp .env.local.example .env.local
   ```

2. [네이버클라우드플랫폼](https://www.ncloud.com/) 콘솔 > AI·NAVER API > Application에서 **Maps**(Web Dynamic Map) 서비스를 사용 설정하고 Client ID를 발급받습니다. Application 등록 시 서비스 URL(`http://localhost:3000`, 배포 도메인)을 반드시 등록해야 지도가 로드됩니다.
3. 장소 검색(위치 선택 화면의 검색창)을 쓰려면 **Geocoding** 서비스도 사용 설정하고 Client ID/Secret을 발급받습니다(지도용 Application과 같아도 되고, 별도 Application이어도 됩니다).
4. `.env.local`에 값을 입력합니다.

   ```
   NEXT_PUBLIC_NAVER_MAP_CLIENT_ID=지도 JS SDK용 클라이언트 ID
   NAVER_MAP_CLIENT_ID=Geocoding용 클라이언트 ID
   NAVER_MAP_CLIENT_SECRET=Geocoding용 클라이언트 시크릿
   ```

   `NAVER_MAP_CLIENT_ID`/`NAVER_MAP_CLIENT_SECRET`에는 절대 `NEXT_PUBLIC_` 접두사를 붙이지 마세요. 그 접두사가 붙은 환경변수는 Next.js가 브라우저에 그대로 노출시키기 때문에, Secret을 서버에만 두려는 목적이 무의미해집니다. 이 두 값은 `app/api/geocode` 라우트(서버)에서만 사용합니다. 지도 JS SDK는 계속 공개용 `NEXT_PUBLIC_NAVER_MAP_CLIENT_ID`만 사용합니다.
5. 개발 서버를 재시작하면 실제 네이버 지도와 장소 검색이 표시됩니다. `NAVER_MAP_CLIENT_SECRET`이 없어도 지도/위치 직접 선택 기능은 정상 동작하고, 검색만 비활성됩니다.

### 지도 로딩 동작 (`src/components/Map`)

- `NEXT_PUBLIC_NAVER_MAP_CLIENT_ID`가 없으면: 곧바로 **Mock Map**을 표시하고, 하단에 "지도 API 미설정: Mock Map으로 표시 중" 문구를 작게 표시합니다.
- 값이 있으면: 네이버 지도 스크립트(`https://oapi.map.naver.com/openapi/v3/maps.js?ncpKeyId=...`)를 `<script>` 태그로 동적 로드합니다. 로딩 중에는 지도 영역에 **"지도를 불러오는 중..."**을 표시하고, `window.naver.maps`가 완전히 준비된 뒤에만 지도를 생성/조작합니다.
- 스크립트 로딩에 실패하면(`onerror`) 자동으로 Mock Map으로 전환되어 앱이 깨지지 않습니다.
- Mock Map도 실제 지도와 동일하게 좌표를 정규화해 ①②③ 순번 마커와 동선(점선)을 그려주고, 마커 클릭/카드 강조가 동일하게 동작합니다.
- 네이버 지도가 자체적으로 붙이는 저작권(© NAVER Corp.)/로고/축척 UI가 다른 화면 요소 위로 튀어나오지 않도록, `Map` 컴포넌트의 지도 wrapper에 `isolate`(새 stacking context 생성) + `overflow-hidden`을 적용했습니다. 이 wrapper는 항상 z-index 0으로 격리되어, 지도 내부 UI가 아무리 높은 z-index를 쓰더라도 Bottom Sheet(`z-[9999]`/`z-[10000]`)나 위치 선택 화면(`z-[10010]`), 일정 리스트(`z-10`) 위로 절대 올라올 수 없습니다.

### 지도 동작 상세

- 일정의 `lat`/`lng`로 커스텀 HTML 마커(①②③ 숫자, 지름 22px)를 그립니다. 경로(Polyline)와 색이 겹쳐 화면이 무거워 보이지 않도록, 마커는 **흰색 배경 + 브랜드 블루(`--primary`) 숫자 + 옅은 회색(#E5E7EB) 테두리**로 표시합니다. 선택된 마커는 배경을 채우는 대신 테두리를 브랜드 블루로 바꾸고(2px) 그림자를 살짝 진하게 하고 1.5배로 커지는 방식으로만 강조해, 파란 면적이 늘어나지 않게 합니다(`NaverMap.tsx`/`MockMap.tsx`).
- 방문 순서대로 Polyline을 연결합니다(선 색상 `#4F7FFF`은 마커와 별개로 그대로 유지합니다).
- 지도 우측 상단의 확대/축소 +/- 버튼(UI)은 표시하지 않습니다(`zoomControl: false`). 마우스 휠, 드래그, 모바일 핀치 확대/축소는 네이버 지도 기본 동작으로 그대로 사용할 수 있습니다.
- 일정 카드를 클릭하면 해당 마커가 강조되고 지도 중심이 그 위치로 부드럽게 이동합니다(`panTo`).
- 장소가 추가/삭제/순서변경되어 목록 자체가 바뀐 경우:
  - 장소가 **1곳**이면 적당한 고정 줌(16)으로 중심만 이동합니다.
  - 장소가 **2곳 이상**이면 항상 `fitBounds`로 모든 장소가 한 화면에 보이도록 자동 확대/축소합니다(상하좌우 80px 여백). 특정 장소만 확대되고 다른 장소가 화면 밖으로 나가는 문제를 막기 위해 고정 줌은 1곳일 때만 사용합니다.
  - `fitBounds` 결과 줌이 지나치게 크거나(장소들이 너무 가까움) 작아지면(너무 멀리 퍼짐) 10~17 범위로 보정합니다.
- 마커를 클릭하면 해당 일정 카드가 선택됩니다. **이미 선택된 마커를 다시 누르면 선택이 해제**되고, **다른 마커를 누르면 기존 선택은 해제되고 새 마커가 선택**됩니다. **지도의 빈 영역을 누르면 선택이 해제**됩니다(마커 클릭은 별도 리스너라 빈 영역 클릭으로 전파되지 않습니다). 선택 해제 시 마커는 원래 크기로, 카드 강조와 메모 펼침도 함께 접힙니다. 네이버 지도/Mock Map 모두 동일하게 동작합니다.
- 순서가 바뀌면 마커 번호와 Polyline 순서도 즉시 갱신됩니다.

## 장소 추가 시 시간순 자동 배치

방문 시간을 입력하면 그 시간에 맞는 자리에 자동으로 끼워 넣습니다(`src/lib/scheduleOrder.ts`의 `insertPlaceByTime`, `OutingScreen.tsx`의 `handleAddPlace`/`handleEditPlace`에서 사용).

- **추가할 때**: 시간이 있으면 기존 카드들 중 "이 장소보다 시간이 늦은 첫 카드" 바로 앞에 삽입됩니다. 시간이 없는 카드는 항상 "더 늦은" 것으로 취급하므로, 시간이 있는 카드는 시간이 없는 카드들보다 항상 위에 옵니다. 시간이 아예 없으면 기존처럼 맨 뒤에 추가됩니다.
- **같은 시간이거나 시간이 없는 카드끼리**는 순서를 건드리지 않습니다 — 새로 추가되는 카드는 같은 시간의 기존 카드들 *뒤에* 붙어서, 먼저 추가한 순서가 유지됩니다.
- **전체를 다시 정렬하는 게 아니라 "끼워 넣기"입니다.** 이미 드래그로 순서를 바꿔둔 카드들은 절대 건드리지 않습니다 — 새 카드 하나만 적절한 위치를 찾아 들어갈 뿐입니다. 그래서 드래그로 정해둔 순서가 이후 장소 추가로 인해 초기화되지 않습니다.
- **수정할 때도 시간이 바뀌면** 같은 방식으로 원래 자리에서 빼서 새 시간 기준으로 다시 끼워 넣습니다. 시간이 안 바뀌었으면(이름/메모/위치만 수정) 자리를 그대로 두어, 같은 시간의 다른 카드와의 순서가 괜히 바뀌지 않게 합니다.
- 이렇게 자리가 정해진 뒤에도 **드래그로 순서를 자유롭게 다시 바꿀 수 있고**, 그 순서가 그대로 유지됩니다(이 배치 로직은 추가/수정 시에만 실행되고, 드래그 결과를 다시 정렬하지 않습니다).
- 순서가 바뀌면 항상 그렇듯 지도 마커 번호와 Polyline도 함께 갱신됩니다(`places` 배열 순서를 그대로 따라가는 기존 로직 그대로).

## 일정 카드 조작

- **순번(①②③) 클릭**: "지도에서 보기" / "네이버지도에서 길찾기" 액션 시트가 뜹니다. 순번 마커는 지도 위 번호 마커와 같은 디자인 시스템(흰 배경 + 브랜드 블루 테두리·숫자)을 사용합니다. 평소엔 얇은 테두리, 선택 시엔 테두리를 굵게(2px) + 그림자로만 강조합니다(`ScheduleCard.tsx`).
  - 지도에서 보기 → 해당 마커 강조 + 지도 중심 이동
  - 네이버지도에서 길찾기 → 저장된 좌표로 `https://map.naver.com/v5/search/{위도},{경도}`를 새 탭으로 엽니다. 사용자가 URL을 직접 입력할 필요가 없습니다(`src/lib/naver.ts`).
- **조작 영역 순서(위→아래)**: 수정(연필) → 위로 이동 → 아래로 이동 → 삭제.
- **수정 버튼(연필 아이콘)**: 누르면 해당 장소의 **수정 Bottom Sheet**가 열립니다(장소명/시간/메모/위치 수정 가능, 기존 값이 채워진 채로 열립니다). 예전에는 이 자리의 아이콘이 드래그 핸들을 겸했는데(점 6개 → 위/아래 화살표 아이콘 순으로 바뀌어왔습니다), 위/아래 화살표 모양이라 "순서 변경 버튼"으로 오해되기 쉬워 지금은 **드래그 기능 없이 수정 전용**(연필 아이콘)으로 분리했습니다.
- **드래그로 순서 변경**: 특정 핸들이 아니라 **카드 전체**가 드래그 대상입니다. 카드를 누른 채로 옮기면(모바일은 길게 눌러서 드래그) dnd-kit으로 순서가 바뀝니다. `MouseSensor`는 6px 이상 움직여야, `TouchSensor`는 150ms 이상 눌러야(5px 이내 흔들림 허용) 드래그로 인식하는 activation constraint 덕분에, 카드 안의 수정/이동/삭제 버튼을 짧게 탭하는 동작이나 목록을 스크롤하는 동작과 서로 간섭하지 않습니다(`ScheduleCard.tsx`의 `motion.div`에 `useSortable`의 `attributes`/`listeners`를 직접 연결).
  - `ScheduleList.tsx`의 센서 구성에 유의하세요: **`PointerSensor`와 `TouchSensor`를 함께 쓰면 안 됩니다.** 최신 브라우저는 터치에도 Pointer 이벤트를 함께 발생시켜서, 거리 기반인 `PointerSensor`와 지연 기반인 `TouchSensor`가 같은 터치를 동시에 잡으려고 경쟁하며 드래그 시작이 몇 초씩 늦어지거나 두 번 시도해야 하는 문제가 있었습니다(실제로 겪은 버그입니다). 터치는 `TouchSensor`에만 맡기고 마우스는 `MouseSensor`로 분리해 해결했습니다.
  - 드래그 시작/카드가 손가락을 따라오는 것/드래그 종료 시 다른 카드가 비켜주는 것 모두 즉시 반응하도록, `isDragging`일 때 카드가 살짝 확대되고(`scale: 1.03`) 그림자가 진해지며(`shadow-xl`), 이 애니메이션과 목록 재배치(`layout`) 애니메이션 모두 짧고(150ms) 스프링이 아닌 `easeOut` transition을 씁니다. 카드에는 `touch-manipulation`을 적용해 모바일 브라우저의 기본 더블탭 확대 감지 지연도 없앴습니다.
  - 놓는 즉시(`onDragEnd`) 새 순서가 `onReorder`로 반영되어 카드·지도 마커 번호·Polyline·LocalStorage가 함께 갱신됩니다.
  - **안내 문구**: "일정 / 총 N곳" 헤더 바로 아래에, 헤더에 딸린 부가 설명처럼 "카드를 드래그하여 순서를 변경해보세요."라는 문구(11px, `#999` 연한 회색)가 뜹니다. 제목과는 `gap-0.5`로 가깝게 붙여 "헤더의 일부"로 읽히게 하고, 문구 자신에 `mb-1.5`(6px) 여백을 더 줘서 그 아래 첫 카드와는 확실히 떨어져 보이도록 했습니다 — 제목 → 안내 문구 → 카드 순으로 정보 계층이 자연스럽게 이어지고, 카드 사이의 "↓ 약 …km · 도보 약 …분" 이동 요약과도 레벨이 명확히 구분됩니다. 장소가 **2곳 이상**일 때만 보이고(1곳뿐이면 순서 개념이 없으니 표시 안 함), 그 외에는 조건 없이 항상 뜨는 **상시 안내**입니다. 기능 자체와는 무관한 안내용 텍스트라 클릭/동작에 영향을 주지 않습니다(`ScheduleList.tsx`).
    - 예전에는 사용자가 드래그로 순서를 한 번이라도 바꾸면 `routie:hasReorderedSchedule` 값을 localStorage에 저장해 그 뒤로는 다시 안 뜨게 했습니다(이미 쓸 줄 아는 사용자에게 계속 노출되는 걸 막기 위함이었음). 이 "한 번 보면 영구히 숨김" 동작 때문에 정작 필요할 때(예: 오랜만에 다시 2곳 이상을 채운 경우) 문구가 안 보인다는 문제가 있어, 지금은 조건을 장소 2곳 이상뿐으로 단순화하고 이 localStorage 플래그와 관련 디버그 로그(`[Routie][Debug][ScheduleList]`)는 완전히 제거했습니다.
- **위/아래 화살표 버튼**: 드래그가 불편한 환경을 위한 보조 수단으로, 수정 버튼 바로 아래 별도 버튼으로 항상 제공됩니다.
- **삭제 버튼**: 카드 삭제, 지도 마커/동선도 즉시 반영됩니다.
- **카드 사이 이동 요약**: 각 카드 사이에 "↓ 약 1.2km · 도보 약 18분"처럼 다음 장소까지의 대략적인 거리/도보 시간을 표시합니다. 실제 길찾기 API가 아닌 두 좌표 간 직선거리(하버사인 공식)를 1km당 15분 기준으로 환산한 예상치이며, 순서가 바뀌면 즉시 다시 계산됩니다(`src/lib/distance.ts`).

## 장소 추가/수정 — 위치 선택 & 저장 장소

일반 사용자가 위도/경도를 직접 알 필요가 없도록, 장소 추가/수정 Bottom Sheet(`PlaceForm.tsx`)에서는 좌표 입력이 기본적으로 숨겨져 있습니다.

**입력 필드 순서(위→아래)**: 위치(지도에서 위치 선택/다시 선택) → 장소명 → 방문 시간 → 메모. 위치 선택 버튼을 맨 위에 둔 이유는, 사용자가 진입하자마자 "위치를 바꾸는 기능"임을 바로 인식할 수 있도록 폼 상단의 대표 액션처럼 보이게 하기 위해서입니다(수정 모드에서 특히 두드러집니다 — 버튼 문구가 "지도에서 위치 다시 선택"으로 바뀌어 있는 채로 맨 위에 옵니다). 즐겨찾기 등록/카테고리 선택은 이 폼 필드 안에 없고, 시트 제목 옆의 별 아이콘으로 분리되어 있습니다(아래 참고).

흐름은 이렇게 이어집니다.

1. **"지도에서 위치 선택"**(이미 위치가 있으면 "지도에서 위치 다시 선택") 버튼으로 위치 선택 화면(`LocationPicker`)을 엽니다 — 장소명을 먼저 입력해두지 않아도 되고, 비워둔 채로 다음 단계에서 자동으로 채워지게 둬도 됩니다
2. 화면 상단 검색창에 원하는 장소(카페, 음식점, 상호명, 지하철역) 또는 지역명을 입력해 검색하거나, 지도를 직접 탭/클릭해 위치를 고릅니다
3. **"이 위치로 선택"**을 누르면 폼으로 돌아와 좌표와 **장소명이 함께** 반영됩니다
4. 필요하면 장소명을 직접 수정하고 장소 추가 완료

- **장소명 자동 입력** (`LocationPicker.tsx`의 `suggestedName`, `PlaceForm.tsx`의 `handleConfirmLocation`): "이 위치로 선택"을 누르면 좌표뿐 아니라 장소명 입력칸도 함께 채워집니다. 항상 **제안일 뿐**이라 강제로 확정되지 않고, 장소명 필드는 그대로 수정 가능한 일반 입력칸입니다.
  - **검색 결과(장소/POI)를 선택한 경우** — 그 결과의 이름을 그대로 씁니다(예: "스타벅스 응암역점", "올리브영"). 역지오코딩을 거치지 않아 빠르고 정확합니다.
  - **지도를 직접 탭하거나(마커 클릭 포함) 지역명 검색 fallback으로 좌표만 얻은 경우** — 이름이 없으므로 `app/api/reverse-geocode`(NCP Maps Reverse Geocoding, `NAVER_MAP_CLIENT_ID/SECRET` 재사용)로 그 좌표 주변 정보를 가져와 아래 우선순위로 이름을 만듭니다.
    1. 건물명/POI명이 있으면 그 이름
    2. 도로명 주소가 있으면 "{도로명 주소} 근처"
    3. 행정동만 있으면 "{행정동} 선택 위치"

  - 지도를 여러 번 연달아 탭해도 항상 **가장 최근 위치의** 역지오코딩 결과만 반영됩니다(이전 요청은 토큰으로 무시).

- **위치 선택은 필수입니다.** 위치를 고르지 않고 "추가하기"를 누르면 "위치를 선택해주세요" 오류가 뜨고 저장되지 않습니다. 예전에는 위치를 안 고르면 `defaultRegion`(지정 외출에서 검색해둔 지역) 또는 서울 좌표에 살짝 오프셋을 줘서 조용히 채워 넣었는데, 이게 실제 데이터 버그의 원인이었습니다 — 그렇게 채워진 좌표가 **진짜 장소처럼 `routie:outings`에 그대로 저장**되어 버려서, "오늘 외출"에 장소가 0개여야 정상 동작하는 현재 위치 자동 반영 로직이 통째로 건너뛰어지고, 지도가 그 "유령 장소"(예: 예전에 검색해뒀던 지역이나 서울시청 좌표)에 계속 고정되는 문제가 있었습니다. 이제는 이런 유령 좌표가 아예 만들어지지 않습니다. **이미 이런 식으로 위치 없이 추가된 장소나 즐겨찾기가 남아있다면(오늘 외출에 기억에 없는 장소가 있거나, 즐겨찾기 목록에 위치를 고른 적 없는 항목이 있다면) 직접 삭제해주세요** — 코드를 고쳐도 이미 저장된 유령 데이터까지 자동으로 지워지지는 않습니다.

- **위치 선택 화면(큰 지도)도 작은 지도와 완전히 같은 위치 결정 로직을 씁니다.** `LocationPicker.tsx`는 `OutingScreen.tsx`이 쓰는 것과 동일한 `NaverMap.tsx`를 그대로 재사용하고, `PlaceForm` → `LocationPicker`로 그 일정에 이미 저장된 다른 장소들(`existingPlaces`, 지금 수정 중인 장소 자신은 제외)과 `isToday`를 전달합니다.

- **위치 선택 화면 검색**: 화면 상단에 "원하는 장소를 입력하거나 지도에서 선택해주세요."라는 안내 문구가 항상 표시되어, 검색은 지도를 이동시키는 용도이고 최종 위치는 지도에서 직접 골라야 한다는 2단계 흐름을 명확히 안내합니다(검색창 placeholder도 동일). 입력 후 검색하면 **서로 다른 두 네이버 API를 순서대로** 호출합니다(`LocationPicker.tsx`) — 동시에 부르지 않고, 장소 검색을 먼저 시도한 뒤 결과가 없을 때만 지역명 검색으로 넘어갑니다.
  1. **장소(POI) 검색 먼저 시도** — `app/api/search-place`가 **네이버 검색 API(지역 검색, 개발자센터 https://developers.naver.com)**를 호출해 카페/음식점/역/상호명 같은 구체적인 장소를 최대 5건 찾습니다(엔드포인트 `https://openapi.naver.com/v1/search/local.json`, 헤더 `X-Naver-Client-Id`/`X-Naver-Client-Secret`, 자격증명 `NAVER_SEARCH_CLIENT_ID`/`SECRET`). "카페", "성수 카페", "강남역", "스타벅스", "올리브영"처럼 지역명만으로는 찾을 수 없는 검색에 씁니다. 결과가 있으면 이름/주소 목록을 검색창 아래에 보여주고, **이 시점엔 아직 좌표로 바꾸지 않습니다** — 이 API가 주는 `mapx`/`mapy`의 좌표계가 계정/버전에 따라 다르게 해석되어 신뢰할 수 없었기 때문입니다(실제로 "카페", "스타벅스" 검색이 반영되지 않는 문제의 원인이었습니다). 제목/카테고리에 붙어 오는 `<b>` 강조 태그는 서버에서 제거해서 내려줍니다.
  2. **목록에서 고르면 그때 지오코딩** — 사용자가 장소 하나를 탭하면, 그 장소의 주소를 `app/api/geocode`(NCP Maps Geocoding API, 헤더 `X-NCP-APIGW-API-KEY-ID`/`X-NCP-APIGW-API-KEY`, 자격증명 `NAVER_MAP_CLIENT_ID`/`SECRET`)에 다시 넣어 최종 좌표를 얻습니다. 좌표는 항상 이 경로로만 얻습니다.
  3. **장소 검색 결과가 없거나 실패하면 지역명 검색으로 자동 대체(fallback)** — 입력한 검색어를 그대로 `app/api/geocode`에 넣어 지도를 이동시킵니다("불광동", "대구"처럼 행정구역 단위 검색이 정상 동작하던 기존 경로 그대로입니다).

  - `PlaceForm`은 `initialValue`가 있으면 기본적으로 "수정 모드"로 간주해 제출 버튼을 "저장하기"로 표시하는데, 즐겨찾기로 미리 채운 경우는 "장소 수정"이 아니라 "장소 추가"이므로 이 문구가 맞지 않습니다. `PlaceForm`에 `submitLabel?: string` prop을 추가해, 이 경우엔 명시적으로 `"추가하기"`를 넘겨줍니다(값을 안 넘기면 기존처럼 `initialValue` 유무로 자동 판단).
  - 사용자가 다른 즐겨찾기를 다시 고르거나 폼을 취소하면 이전에 채워둔 값이 남지 않도록, `PlaceForm`에 `key={favoriteDraft?.id ?? "manual"}`를 줘서 즐겨찾기가 바뀔 때마다 폼이 완전히 새로 마운트되며 필드가 초기화됩니다.
- **즐겨찾기 아이콘은 서비스 전체에서 북마크로 통일**되어 있습니다 — 아웃라인 북마크는 저장되지 않은 상태, 채워진 북마크(`fill-primary text-primary`)는 저장된 상태입니다(예전에는 별 아이콘이었습니다). 홈 화면의 "즐겨찾기" 메뉴 카드 아이콘도 같은 이유로 북마크로 바꿨습니다. "저장한 장소 불러오기" 버튼(장소 추가 시트 상단)은 원래부터 아웃라인 북마크였어서 그대로입니다 — 특정 장소의 저장 여부를 나타내는 토글이 아니라 "여기서 불러올 수 있다"는 진입점이라, 계속 채워지지 않은 모양입니다.
- **"장소 추가"/"장소 수정" 시트 제목 옆 즐겨찾기 북마크** (`FavoriteBookmarkButton.tsx`, `BottomSheet`/`Sheet`의 `titleAction` prop으로 제목과 같은 줄, 우측 상단에 배치): 두 시트 모두 아웃라인/채워진 북마크 아이콘이 있습니다. 아직 즐겨찾기가 아닐 때(아웃라인) 누르면 **카테고리 선택 Bottom Sheet**(`CategoryPickerSheet.tsx`)가 열리고, 칩을 골라 "확인"을 누르면 그 카테고리로 즐겨찾기에 등록됩니다(채워진 모양으로 바뀜). 이미 즐겨찾기일 때(채워짐) 누르면 카테고리를 다시 묻지 않고 바로 제거됩니다(아웃라인으로 바뀜).
  - **"장소 수정"** (`FavoriteToggleButton.tsx`): 이미 저장된 장소라 좌표(위도/경도) 일치로 즉시 즐겨찾기 여부를 판단합니다 — `Place`와 `FavoritePlace`는 서로 다른 id 체계라(즐겨찾기로 저장할 때 항상 새 id 발급) 직접 연결하는 필드가 없어서, "같은 위치"를 기준으로 매칭합니다. 북마크를 누르는 즉시(폼 제출과 무관하게) 반영됩니다.
  - **"장소 추가"** (`OutingScreen.tsx`의 `pendingFavoriteCategory`): 아직 저장되지 않은 장소라 좌표로 판단할 대상이 없습니다. 대신 북마크를 눌러 카테고리를 고르면 "제출하면 이 카테고리로도 즐겨찾기에 넣겠다"는 의도만 임시로 들고 있다가, **"추가하기"를 눌러 장소가 실제로 저장되는 순간** 그 확정된 이름/좌표로 즐겨찾기도 함께 생성합니다. 시트를 취소하거나 "저장한 장소 불러오기"로 전환하면 이 의도는 초기화됩니다(엉뚱한 장소에 잘못 붙는 것을 막기 위함). 이미 골라둔 카테고리가 있는 채로 시트를 다시 열면 그 칩이 미리 선택되어 있습니다(`initialCategory` prop).
- **카테고리 선택 Bottom Sheet** (`CategoryPickerSheet.tsx`): 위 두 북마크 버튼이 공유하는 화면이자, `TimePicker`와 같은 "고르는 동안은 미리보기, 확인을 눌러야 확정" 패턴을 씁니다.
  - 카테고리는 **칩(chip)** 형태로 가로로 나열됩니다(`flex flex-wrap`). 칩을 누르면 선택되어 브랜드 컬러(`bg-primary text-primary-foreground`)로 채워지고, 다른 칩을 누르면 그쪽으로 선택이 옮겨갑니다(한 번에 하나만 활성화). 선택되지 않은 칩은 기본 스타일(`border-border bg-white text-muted-foreground`)입니다.
  - 각 칩 오른쪽 위에 작은 **X** 버튼이 있어 기본 카테고리를 포함한 모든 카테고리를 삭제할 수 있습니다. 삭제 전 확인은 `ConfirmDialog.tsx`로 뜨는 **화면 중앙 모달**입니다(칩 목록이 있는 Bottom Sheet 레이어와는 명확히 구분되는 별도 레이어) — 삭제하면 그 카테고리를 쓰던 기존 즐겨찾기는 "기타"로 자동 변경됩니다(즐겨찾기 자체는 지워지지 않음). 확인 버튼은 빨간 `destructive`가 아니라 **브랜드 메인 컬러(파란색, 기본 `Button` variant)** 로 강조되어 있고, 취소 버튼은 기존 `outline` 스타일 그대로입니다 — 과도한 경고성 톤 대신 Routie의 미니멀한 톤을 유지하기 위함입니다. 지금 선택 중인 칩을 삭제하면 선택이 풀립니다.
  - 칩 목록 끝의 점선 원형 **+** 버튼으로 새 카테고리를 추가할 수 있습니다. 추가하면 기존 칩과 똑같은 모양으로 목록에 나타나고 곧바로 선택 상태가 되어, 바로 "확인"만 누르면 됩니다.
  - 아무 칩도 선택하지 않은 상태에서는 "확인" 버튼이 비활성화됩니다.
  - `useCategories.ts`에는 `renameCategory`(이름 바꾸기)도 구현되어 있지만, 이번 칩 UI에는 별도의 이름 수정 버튼을 넣지 않았습니다 — 이름을 바꾸고 싶으면 삭제 후 원하는 이름으로 다시 만들거나, 필요하면 이 시트에 편집 버튼을 추가로 붙일 수 있습니다.
  - **기본 카테고리는 맛집/카페/가볼 곳 3개**입니다(`types/category.ts`의 `DEFAULT_CATEGORIES`). 예전 기본값(카페/병원/음식점/전시·문화/쇼핑/기타)에서 바뀐 것으로, 이미 `routie:categories`가 저장되어 있는 브라우저에는 영향이 없습니다(최초 1회 시딩값일 뿐이라 기존 저장값을 덮어쓰지 않습니다).
  - "장소 추가"/"장소 수정" 시트가 이미 열려 있는 상태에서 그 위에 겹쳐 열리는 **중첩 Dialog**입니다 — `LocationPicker`가 이미 같은 구조로 검증되어 있어(위 "지도에서 위치 선택" 절 참고), 같은 `BottomSheet` 컴포넌트를 그대로 재사용해도 레이어링이 올바르게 동작합니다.
- **삭제 확인 모달** (`ConfirmDialog.tsx`): `Sheet`처럼 아래에서 슬라이드되는 형태가 아니라, `LocationPicker.tsx`와 같은 방식으로 Radix `Dialog.Root`를 직접 써서 **화면 정중앙에 고정된 카드**로 뜨는 범용 확인 모달입니다(현재는 카테고리 삭제 확인에만 쓰이지만 `title`/`description`/`confirmLabel`/`onConfirm` props로 재사용 가능). `CategoryPickerSheet`처럼 이미 열려 있는 Bottom Sheet 위에 겹쳐 뜰 수 있어야 하므로 `LocationPicker`와 동일하게 `data-routie-overlay` 마커를 오버레이/콘텐츠 양쪽에 붙였고(안 그러면 `ui/sheet.tsx`의 바깥-클릭 감지가 이 모달 안의 클릭을 "Sheet 바깥 클릭"으로 오인해 아래 시트까지 닫아버립니다), z-index도 `z-[10020]`/`z-[10021]`로 일반 Sheet(9999/10000)와 `LocationPicker`(10010)보다 위에 두어 어떤 화면 위에서 열리든 항상 최상단에 보이도록 했습니다. 확인 버튼은 기본(un-set) `Button` variant라 브랜드 블루이고, 취소 버튼은 `outline`입니다.
  - **중앙 정렬 구조**: `Dialog.Content`의 자식은 transform을 전혀 쓰지 않는 순수 `fixed inset-0 flex items-center justify-center` div이고, 실제 애니메이션이 붙는 카드(`motion.div`, `initial/animate/exit`로 opacity+scale)는 그 안의 자식입니다. 처음엔 `left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2`로 카드 자체를 중앙 정렬했다가, Framer Motion이 `scale` 애니메이션을 걸 때 그 요소의 인라인 `transform` 스타일을 통째로 재계산해서 덮어써 버려 `translate(-50%,-50%)` 오프셋이 사라지고 카드가 화면 왼쪽으로 치우쳐 보이는 버그가 있었습니다 — position(centering)과 animation(scale) 책임을 서로 다른 요소로 분리해서 고쳤습니다. (transform 기반 utility class와 Framer Motion의 transform 애니메이션을 같은 요소에 같이 쓰지 않는 것이 일반적인 규칙입니다.)
- **Bottom Sheet 내부 스크롤 + 드래그로 높이 조절** (`ui/sheet.tsx`): 모든 `BottomSheet`(장소 추가/수정, 카테고리 선택, 홈 화면 메뉴 등)에 공통 적용됩니다.
  - 시트 내부는 **헤더(드래그 핸들 + 제목, 고정)** 와 **콘텐츠(세로 스크롤)** 두 영역으로 나뉩니다. 콘텐츠 영역은 `min-h-0 flex-1 overflow-y-auto`라 내용이 많아져도 실제로 스크롤되며, 마지막 요소(예: 추가하기 버튼)까지 항상 접근할 수 있습니다. 예전에는 `{children}`이 감싸는 div 없이 `max-h-[88vh] flex flex-col` 컨테이너에 바로 들어가 있어서, `flex` 자식은 기본적으로 내용 크기만큼 자라려 하고(`min-height: auto`) 부모의 `overflow`도 지정돼 있지 않았던 탓에 내용이 길어지면 시트 바깥으로 넘쳐 잘려 보이는 문제가 있었습니다.
  - 시트 높이는 기본 `88vh`, 확장 시 `94vh`로 Framer Motion `animate`의 `maxHeight` 값을 상태(`isExpanded`)에 따라 애니메이션합니다. 시트를 열 때마다 `isExpanded`는 항상 `false`로 초기화됩니다(이전에 확장한 채로 닫아도 다음엔 기본 높이로 시작).
  - **드래그 핸들**(제목 위 알약 모양)을 위/아래로 끌면 높이가 바뀝니다 — 위로 `48px` 이상 끌면 확장(`94vh`), 아래로 `48px` 이상 끌면 이미 확장된 상태면 기본 높이로 축소, 기본 높이에서 더 끌면 시트가 닫힙니다. 실제 알약 모양(`h-1.5 w-10`)은 그대로 두고, 드래그로 잡을 수 있는 영역만 `h-5 w-16`으로 더 넉넉하게 잡아 터치하기 쉽게 했습니다. 핸들은 `dragConstraints={{top:0,bottom:0}}` + `dragElastic`으로 살짝 튕겼다가(고무줄 느낌) 제자리로 돌아오는, Framer Motion에서 흔히 쓰는 "고무줄 드래그" 패턴입니다 — 실제 드래그 방향/거리는 `onDragEnd`의 `info.offset.y`로 판단하고, 핸들 자체 위치는 매번 원점으로 복귀합니다.
  - 드래그 제스처는 **핸들 요소에만** 걸려 있고(제목·즐겨찾기 버튼이 있는 줄은 별도 div), 스크롤은 **콘텐츠 div에만** 걸려 있어 — 서로 다른 DOM 요소이므로 핸들 드래그와 콘텐츠 스크롤 제스처가 충돌하지 않습니다.

## 자동 저장

- 로그인/서버 없이 **LocalStorage에만** 저장됩니다. 장소 추가/수정/삭제/순서 변경/메모/일정 제목은 물론, `useLocalStorage`를 쓰는 모든 데이터(오늘·지정 외출의 장소 목록, 즐겨찾기, 카테고리, 기본 지역)가 값이 바뀔 때마다 즉시 저장됩니다(`src/hooks/useLocalStorage.ts`). 별도의 "저장" 버튼은 없습니다 — 앱을 끄거나 새로고침해도 브라우저의 사이트 데이터를 지우지 않는 한 그대로 남아 있습니다.
- 외출 화면(오늘/지정 외출) 오른쪽 하단에 **"마지막 저장: N분 전"** 같은 작은 회색 문구가 뜹니다(`SaveStatusIndicator.tsx`). FAB(➕)와 같은 높이·같은 하단 여백(`max(1.5rem, safe-area)`)에 두되, FAB 왼쪽으로 0.75rem 떨어뜨려 겹치지 않게 배치했습니다. `useLocalStorage`가 실제로 `localStorage.setItem`을 호출할 때마다 `src/lib/saveStatus.ts`의 전역 저장소에 시각이 기록되고, 이 컴포넌트가 그 값을 구독해 상대 시간으로 보여줍니다(`src/lib/time.ts`의 `formatRelativeTime`: 방금 전 → N분 전 → N시간 전 → 어제 → N일 전 → 날짜, 30초마다 갱신). 저장이 아직 한 번도 없었다면 표시되지 않고, 처음 나타날 때는 페이드인으로 자연스럽게 등장합니다. 클릭 동작이 없는 안내 문구라 `pointer-events-none`으로 두어 아래 카드 터치를 가리지 않습니다.

## 홈 화면 — 최근 작성한 일정

- 홈 화면 하단에 **"최근 작성한 일정"** 섹션이 있습니다(`src/app/page.tsx`, `RecentOutingCard.tsx`). 장소가 1곳 이상 있는 일정만, **마지막 저장이 최신인 순서**로 카드 목록을 보여줍니다(`useRecentOutings`, `src/hooks/useOutings.ts`).
- 각 카드는 **일정 제목 + "마지막 저장: N분 전" + 장소 수**를 보여주고, 카드 본문(아이콘+텍스트 영역)을 누르면 해당 날짜의 외출 화면(`/outing/{date}`)으로 바로 이동해 이어서 편집할 수 있습니다. 홈 화면 상단의 기존 메뉴 카드(오늘 외출/지정 외출/즐겨찾기)와 같은 카드 톤(테두리 + `bg-card` + `shadow-sm`, 원형 아이콘)을 그대로 씁니다.
- **외출 화면 헤더 정보 계층**: 날짜가 헤더에서 가장 눈에 띄는 정보입니다 — 굵은 큰 글씨(`text-lg font-bold`)로 가운데 정렬해 가장 먼저 읽히고, 그 아래에 **날씨 정보**(`WeatherBadge.tsx`, 아래 "날씨 정보" 절 참고)가 작은 글씨로 보조 정보처럼 뜹니다(`OutingHeader.tsx`).
  - **일정 제목 수정 UI는 현재 헤더에 없습니다.** 예전에는 날짜 아래 문구("오늘 외출은 어디로?")를 탭해 제목을 설정할 수 있었는데, 그 자리에 날씨 정보를 넣기로 하면서 문구와 함께 탭-투-수정 진입점도 제거했습니다. 제목 데이터 자체(`OutingEntry.title`)와 이를 읽고 쓰는 `useOuting()`의 `title`/`setTitle`은 그대로 남아 있어 나중에 다른 진입점(예: 메뉴)으로 다시 연결할 수 있지만, 지금은 어디서도 호출되지 않습니다.
  - **홈 화면 "최근 작성한 일정" 카드**의 제목은 `useOutings.ts`에서 커스텀 제목이 없으면 "{날짜} 외출"(예: "7월 8일 (수) 외출")로, 있으면 그 실제 제목 텍스트를 그대로 보여줍니다 — 다만 지금은 제목을 설정할 UI가 없으므로 사실상 항상 날짜 기반 제목만 보이게 됩니다.
- **지정 외출(달력) 바로가기**: 헤더 오른쪽 끝에 달력(Outline) 아이콘이 항상 뜨고, 누르면 `/calendar`(지정 외출 화면)로 이동합니다(`OutingHeader.tsx`). 아이콘 자체는 다른 헤더 아이콘과 같은 크기(`h-5 w-5`)를 쓰지만, 버튼(터치 영역)은 모바일에서 누르기 쉽도록 높이 `h-10`(40px)로 다른 아이콘 버튼들보다 넉넉하게 잡았습니다. `router.push`로 이동하므로 뒤로 가기를 누르면 다시 원래 보고 있던 외출 화면으로 돌아옵니다.
  - 원래는 `isToday`일 때만 보이고 다른 날짜에서는 같은 너비의 빈 자리로 대체하도록 만들었는데, 좌우 화살표로 날짜를 넘기면 아이콘이 사라지는 버그가 있었습니다(오늘이 아닌 모든 날짜에서 사라짐). 지금은 `isToday` 조건 없이 날짜와 무관하게 항상 렌더링해, 이전/다음 날짜로 이동해도 헤더 레이아웃이 그대로 유지됩니다.
  - **"오늘" 표시는 별개의 요소입니다**: 오늘 날짜일 때만 이 아이콘 왼쪽에 "오늘" 알약(pill)이 함께 뜨지만, 아이콘과 하나로 묶인 버튼이 아니라 각자 독립된 요소입니다.
    - **"오늘" 알약**: 배경 `bg-accent`(연한 파랑) + 텍스트 `text-accent-foreground`(진한 파랑) + `rounded-full`. 클릭 동작이 없는 순수 상태 표시(`<span>`)입니다 — "지금 오늘 날짜를 보고 있다"는 사실만 알려줍니다.
    - **달력 아이콘 버튼**: 배경 없이(호버 시에만 `hover:bg-muted`) 검정에 가까운 `text-foreground` 아웃라인 아이콘만 표시하고, 누르면 `/calendar`로 이동합니다. `isToday`와 무관하게 항상 같은 모양(`h-10 w-10` 원형)입니다.
    - 두 요소는 `gap-1.5`를 둔 같은 flex 줄에 나란히 있을 뿐, 한쪽을 눌러도 다른 쪽 동작에 영향을 주지 않습니다.
- **카드 오른쪽 위 더보기(⋮) 메뉴**: 카드 본문과 별개의 버튼이라 눌러도 이동하지 않습니다(`RecentOutingCard.tsx`). 누르면 기존 "지도에서 보기/네이버지도에서 길찾기" 액션 시트와 같은 형태의 Bottom Sheet가 뜨고, 세 가지를 고를 수 있습니다(`src/app/page.tsx`).
  1. **이어서 작성** — 카드 본문을 누른 것과 동일하게 `/outing/{date}`로 이동합니다.
  2. **일정 복제** — 장소 목록(각 장소는 새 id로 재발급)과 제목("원본 제목 (복사본)")을 그대로 복사해 **가장 가까운 빈 날짜**(내일부터 순서대로, 이미 장소가 있는 날짜는 건너뜀)에 새 일정으로 저장합니다(`duplicateOuting`, `useOutings.ts`). 일정은 날짜당 하나만 존재할 수 있는 구조라 복제본도 반드시 어떤 날짜엔가는 있어야 하기 때문입니다. 복제 직후 "OO 일정을 복제했어요" 토스트가 뜨고, 복제본은 방금 저장되었으므로 목록 맨 위에 나타납니다.
  3. **일정 삭제** — 바로 지우지 않고 확인 Bottom Sheet("정말 이 일정을 삭제하시겠습니까?" / "삭제된 일정은 복구할 수 없습니다." / 취소·삭제 버튼)를 먼저 띄웁니다. **삭제**를 눌러야 실제로 지워지고(`deleteOuting`), 지운 뒤에는 복구할 방법이 없습니다.
- 작성한 일정이 하나도 없으면 카드 목록 대신 **"아직 작성한 일정이 없습니다."** 안내와 **"새 일정 만들기"** 버튼(오늘 외출로 이동)을 보여줍니다.
- 일정 데이터 구조가 `Place[]`(장소 배열만)에서 `{ title, places, updatedAt }`(제목/장소/마지막 저장 시각)로 바뀌었습니다. 이전 버전에서 이미 저장해둔 데이터(배열 형태)를 읽을 때는 제목 없음·저장시각 0으로 자동 변환해서 기존 일정을 잃지 않습니다(`useOutings.ts`의 `normalizeEntry`).

## 날씨 정보 (기상청 동네예보 API 연동)

외출 화면 헤더의 날짜 아래에 날씨 아이콘 + 현재 기온 + 외출 관련 한 줄 코멘트가 뜹니다(`WeatherBadge.tsx`).

- **위치**: `navigator.geolocation.getCurrentPosition`으로 실제 현재 위치(GPS 위도/경도)를 우선 사용합니다(`enableHighAccuracy: false` — 날씨는 기상청 5km 격자 단위라 GPS급 정밀도가 필요 없고, `maximumAge: 10분`이라 그 안에 캐시된 위치도 그대로 씁니다). **위치 권한이 거부되었거나, geolocation API 자체를 지원하지 않거나, 조회에 실패하면** `types/place.ts`의 `DEFAULT_COORDS`(서울시청, 위경도 기준 기상청 격자로는 nx=60/ny=127)로 즉시 대체합니다.
  - **클라이언트 진단 로그** (`[Routie][WeatherBadge]`, `WeatherBadge.tsx`): `[GPS]` 권한 허용/조회 성공 또는 거부/실패(→서울 대체) 여부, `[위치 확정]` 최종 사용 좌표와 `source`("gps" 또는 "fallback"), `[/api/weather 호출]` 성공/실패(status), `[최종 날씨 데이터 수신]` 성공/실패를 각각 구분해서 남깁니다. 이 `source` 값은 `/api/weather?...&source=gps|fallback`로 서버에도 그대로 전달되어, 서버 로그의 `[STEP 2/5]`에서도 이 요청이 GPS 좌표인지 fallback 좌표인지 바로 확인할 수 있습니다(동작에는 영향 없는 순수 진단용 값).
- **API 라우트** (`src/app/api/weather/route.ts`, Next.js Route Handler): 클라이언트는 이 라우트만 호출하고, 실제 기상청 인증키(`KMA_API_KEY`)는 여기(서버)에서만 `process.env.KMA_API_KEY`로 읽습니다 — `NEXT_PUBLIC_` 접두사가 없어 브라우저 번들에 절대 포함되지 않습니다. 요청/응답을 `[Routie][weather]` 로그로 남기되, 키 값은 `geocode` 라우트와 같은 방식으로 앞 4자리만 마스킹해서 출력합니다.
- **기상청 API허브(apihub.kma.go.kr)의 `nph-dfs_vsrt_grd`(초단기예보 격자)를 사용합니다.** ⚠️ **data.go.kr(공공데이터포털)도 아니고, API허브의 `VilageFcstInfoService_2.0`(JSON REST) 계열도 아닙니다** — 이 프로젝트는 활용신청이 **"예특보 > 단기예보 > 동네예보 > `/cgi-bin/url/nph-dfs_vsrt_grd`"** 로 승인되어 있어, 반드시 이 legacy 텍스트 API(`https://apihub.kma.go.kr/api/typ01/cgi-bin/url/nph-dfs_vsrt_grd`)를 호출해야 합니다. 인증은 쿼리 파라미터 `authKey`로 전달합니다(헤더 아님).
  - **응답 형태가 완전히 다릅니다.** JSON이 아니라, **한 번의 호출 = 한 변수(T1H 또는 SKY 또는 PTY 또는 RN1) 하나에 대한 전국 149(동서, nx) × 253(남북, ny) 격자 전체**를 콤마로 구분된 숫자 텍스트로 돌려줍니다(한 줄에 최대 20개씩, 행마다 8줄로 줄바꿈됨. `-99.00`은 "데이터 없음"). `vars`에 쉼표로 여러 변수를 한 번에 넣는 것은 **지원되지 않습니다**(실제로 호출해서 확인함 — `vars=T1H,SKY,PTY,RN1`처럼 넣으면 대부분 `-99`로 깨진 응답이 돌아옵니다). 그래서 T1H/SKY/PTY/RN1을 **각각 별도로, `Promise.all`로 병렬 호출**합니다(`fetchGridVariable`) — 순차 호출하면 legacy 엔드포인트 특성상 한 번에 수십 초(첫 호출 기준 15~45초, 이후 같은 `tmfc`는 캐시되어 1초 이내)씩 걸려 총 대기시간이 배로 늘어나기 때문입니다.
  - **한 지점의 값을 뽑는 "격자 좌표 → 배열 인덱스" 매핑은 실제로 살아있는 authKey로 호출해서 검증했습니다**(`src/lib/kmaGrid.ts`의 `parseKmaGridCell`): 텍스트의 행 순서는 `ny` 오름차순으로 그대로 대응합니다(뒤집혀 있지 않음) — 부산(nx=98,ny=76)과 제주(nx=53,ny=38) 좌표를 이 규칙으로 뽑으면 실제 기온 값이 나오고, 행을 뒤집어서 뽑으면 두 도시 모두 격자 밖(-99)으로 나가버리는 것으로 확인했습니다(부산/제주는 실제 육지·해안 도시라 값이 있어야 정상).
  - **`vars` 종류는 딱 4개, T1H(기온)/SKY(하늘상태)/PTY(강수형태)/RN1(1시간 강수량)만 요청합니다.** `VilageFcstInfoService_2.0`(JSON REST) 계열이나 `getUltraSrtNcst`는 여전히 코드에서 전혀 호출하지 않습니다. (단기예보 `nph-dfs_shrt_grd`는 이제 아래 "단기예보(지정 외출) 연동"에서 별도로 호출합니다 — 초단기예보 흐름 자체는 이 항목 그대로입니다.)
  - **`tmfc`(발표시각) 계산**(`route.ts`의 `getTmfc`): 초단기예보는 매시 30분에 생성되어 45분부터 조회 가능합니다. 서버가 어떤 타임존에서 돌든(Vercel 배포 환경은 UTC) 항상 KST(UTC+9) 기준으로 계산하도록, 로컬 타임존 오프셋을 걷어낸 뒤 KST 오프셋(+9시간)을 다시 더합니다. 현재 분이 45분 미만이면 아직 이번 시간 예보가 준비되지 않았을 수 있어 한 시간 전 `tmfc`(예: `202607091830`, YYYYMMDDHHmm)를 씁니다.
  - **authKey는 항상 `encodeURIComponent`로 인코딩해서 URL에 붙입니다** — 특수문자가 섞여 있어도 안전하도록 하는 방어적 조치입니다.
  - **에러 응답 형태도 legacy 스타일입니다.** 파라미터가 잘못되면 JSON/XML이 아니라 `# input variable error (-1)` 같은 한 줄 주석을 돌려줍니다 — `parseKmaGridCell`이 응답이 `#`으로 시작하면 이를 감지해 에러로 처리합니다.
  - **진단 로그** (`[Routie][weather]`): 변수별로 요청 URL(마스킹된 authKey 포함)과 응답 status/본문 앞부분, 격자 파싱 성공/실패 여부를 모두 남깁니다. **단계 구분**: `[STEP 1/5]` KMA_API_KEY 로딩 → `[STEP 2/5]` 좌표→격자(nx/ny)/tmfc 변환 → `[STEP 3/5]` 변수별 API허브 요청/응답(4개 병렬) → `[STEP 4/5]` 변수별 격자 파싱 → `[STEP 5/5]` 최종 데이터 반환. **위치(GPS)와 API 연동 문제를 분리해서 진단하려면**: `curl "http://localhost:3000/api/weather?lat=37.5665&lng=126.978"`처럼 서울 고정 좌표로 직접 호출해보세요 — GPS를 전혀 거치지 않으므로, 여기서 [STEP 1/5]에서 멈추면 API 키 문제, [STEP 3/5]~[STEP 4/5]에서 멈추면 활용신청/파싱 문제입니다.
- **아이콘 + 코멘트 매핑** (`src/lib/weatherPresentation.ts`의 `describeWeather(sky, pty, rn1)`): 강수형태(PTY)가 "없음"이 아니면 하늘상태(SKY)보다 강수를 우선합니다 — 비/눈이 오는데 "맑음" 아이콘이 뜨면 오히려 오해를 주기 때문입니다. 예: ☀️ 맑음 "외출하기 좋은 날이에요.", 🌧️ 비 "비가 와요. 우산을 챙기세요.", ❄️ 눈 "눈이 내려요. 미끄럼에 주의하세요."
  - **RN1(1시간 강수량, mm 단위 숫자)은 PTY가 "0"(강수없음)일 때도 보조 신호로 확인합니다.** 강수 경계 시점에는 PTY와 RN1이 살짝 어긋나는 경우가 있어(PTY는 아직 "0"인데 RN1엔 이미 소량의 강수량이 잡히는 경우), RN1이 0보다 크면 비가 오는 것으로 취급해 비 아이콘/코멘트를 보여줍니다(격자의 "데이터 없음" 값 `-99`는 자연히 0보다 작아 이 조건에 걸리지 않습니다). RN1 값 자체(mm 수치)를 화면에 노출하지는 않고, 판단 근거로만 씁니다.
- **한 줄 레이아웃**: 로딩/성공/실패 세 상태 모두 `<p className="truncate text-xs text-muted-foreground">` 한 줄로 통일되어 있습니다(`WeatherBadge.tsx`) — 헤더가 상태에 따라 커지고 줄어드는 것을 막기 위함입니다. 로딩 중 "☀️ 날씨 정보를 불러오는 중입니다.", 실패 시 "☁️ 날씨 정보를 가져올 수 없습니다.", 성공 시 **"{아이콘} {기온}℃ · {코멘트}"**(예: "🌧️ 24℃ · 비가 와요. 우산을 챙기세요.") 형태로 아이콘/기온/코멘트가 모두 한 행에 `·` 구분자로 표시됩니다. 날짜(`text-lg font-bold`)는 계속 헤더에서 가장 크고 눈에 띄는 정보로 유지되고, 날씨는 그 아래 보조 정보 크기(`text-xs`)를 유지합니다. 화면에는 문구 하나만 보이지만, `/api/weather`가 실패 응답을 주면 브라우저 콘솔에 `[Routie][WeatherBadge] /api/weather 호출 실패`로 실제 `status`와 응답 본문을 그대로 남겨 원인을 바로 확인할 수 있습니다.
  - **첫 호출은 최대 수십 초까지 걸릴 수 있습니다** — `nph-dfs_vsrt_grd`가 매 `tmfc`마다 전국 격자를 새로 계산해 내려주는 legacy 엔드포인트라 그렇습니다(실측 15~45초). 같은 `tmfc`로 다시 요청하면(같은 시간대 내 재방문 등) API허브 쪽에 캐시되어 1초 이내로 빨라집니다. 로딩 문구가 평소보다 오래 떠 있는 것은 정상입니다.
- **환경변수 설정**: `.env.local`에 `KMA_API_KEY=` 를 추가하고, [기상청 API허브](https://apihub.kma.go.kr/)에서 회원가입 후 발급받은 **authKey**를 값으로 넣으세요(**data.go.kr 서비스키가 아닙니다**). 그 뒤 API허브에서 정확히 **"예특보 > 단기예보 > 동네예보 > 초단기예보(`nph-dfs_vsrt_grd`)"** 를 **활용신청**해야 실제로 호출이 됩니다 — 같은 "동네예보" 안에도 단기예보/초단기예보/실황이 각각 별도로 활용신청해야 하는 항목으로 나뉘어 있으니, 다른 항목이 아니라 정확히 이 항목인지 확인하세요. **Next.js는 `.env.local`을 dev 서버 실행 중에도 감지해 자동으로 다시 읽어들이지만(`Reload env: .env.local` 로그), 확실하게 반영하려면 서버를 재시작하는 것을 권장합니다.** 키가 비어 있거나 활용신청이 안 되어 있으면 위 실패 상태 문구가 계속 뜰 뿐 앱의 다른 기능에는 영향이 없습니다.

### 단기예보(지정 외출) 연동

오늘 외출은 위에서 설명한 초단기예보(`nph-dfs_vsrt_grd`)를 그대로 씁니다. **지정 외출(오늘이 아닌 날짜)은 그 날짜에 맞는 단기예보(`nph-dfs_shrt_grd`)를 대신 사용합니다** — 초단기예보는 "지금으로부터 몇 시간"만 예보하는 상품이라 다른 날짜에는 애초에 값이 없기 때문입니다. **같은 `KMA_API_KEY`(authKey)를 그대로 재사용**하며, 새 환경변수는 추가되지 않았습니다.

- **분기 지점**: `WeatherBadge`가 `isToday`가 아니면 `/api/weather` 요청에 `&date=YYYY-MM-DD`를 추가로 붙입니다. `route.ts`의 `GET`은 이 `date` 파라미터 유무만으로 초단기예보/단기예보 두 흐름을 분기합니다 — `date`가 없으면(오늘 외출) 기존 로직이 **한 글자도 바뀌지 않은 채** 그대로 실행되고, `date`가 있으면 완전히 별도의 함수(`handleShortTermForecast`)로 처리됩니다. 두 흐름은 각자 실패해도 서로에게 영향을 주지 않습니다.
- **발표시각(`tmfc`) 계산**: 단기예보는 하루 8번(02/05/08/11/14/17/20/23시)만 발표됩니다. "발표 시각 + 10분"이 지난 가장 최근 발표를 씁니다(정확한 지연 시간은 초단기예보의 "30분 생성/45분 조회"처럼 문서로 확인하지 못해 보수적으로 10분을 잡았습니다 — 아래 "검증됨/검증 필요" 참고).
- **예보시각(`tmef`)**: 지정 외출은 시간 단위 개념이 없는 "하루 전체" 일정이라, 그 날짜의 **정오(12:00)**를 대표 시각으로 골라 `tmef=YYYYMMDD1200`을 요청합니다(`toTmef`).
- **조회 가능 범위(최대 3일)**: 단기예보는 발표 시점부터 최대 3일 뒤까지만 제공됩니다(`SHRT_MAX_DAYS_AHEAD`). 선택한 날짜가 이미 지났거나 이 범위를 벗어나면, API를 호출하지도 않고(쿼터 절약) 바로 `reason: "OUT_OF_FORECAST_RANGE"`로 응답합니다 — 화면에는 기존과 같은 "☁️ 날씨 정보를 가져올 수 없습니다." 문구가 뜹니다.
- **변수**: TMP(기온)/SKY(하늘상태)/PTY(강수형태)/PCP(강수량)를 초단기예보의 T1H/SKY/PTY/RN1 자리에 각각 대응시켜, 같은 `describeWeather()`/아이콘·코멘트 매핑을 그대로 재사용합니다. 격자 파싱(`parseKmaGridCell`)도 초단기예보와 동일한 전국 149×253 텍스트 격자 포맷이라고 가정하고 그대로 재사용합니다.
- ✅ **실제 승인된 authKey로 검증됨**: 위 endpoint 이름(`nph-dfs_shrt_grd`)·파라미터(`tmfc`+`tmef`+`vars`+`authKey`)·변수 코드(TMP/SKY/PTY/PCP)·격자 응답 포맷 전부 실제 호출로 확인했습니다(예: `curl "http://localhost:3000/api/weather?lat=37.5665&lng=126.978&date=2026-07-11"` → 실제 기온/하늘상태 값이 정상적으로 돌아옴). ⚠️ 다만 **`tmfc`의 "발표 후 10분" 지연 가정과, 3일이라는 최대 조회 범위**는 공식 문서로 재확인하지 못한 값입니다 — 발표 시각 직후(예: 02:05)나 D+3 경계 날짜에서 실패하면 이 두 상수(`SHRT_ISSUE_HOURS`의 지연 분, `SHRT_MAX_DAYS_AHEAD`)부터 의심해보세요.
- **로그**: 단기예보 흐름은 `[STEP 3/5(단기)]`/`[STEP 4/5(단기)]` 접두사로 초단기예보 로그(`[STEP 3/5]`/`[STEP 4/5]`)와 구분됩니다.

## 데이터 구조 (날짜별 저장)

```ts
type Place = {
  id: string;
  name: string;
  time?: string;
  memo?: string;
  lat: number;
  lng: number;
};

// LocalStorage key: "routie:outings"
type OutingEntry = {
  title: string | null; // null이면 홈/헤더에 날짜 기반 기본 제목을 표시
  places: Place[];
  updatedAt: number; // 마지막 저장 시각(ms) — "최근 작성한 일정" 정렬/"마지막 저장" 문구에 사용
};
type OutingMap = Record<string, OutingEntry>; // { "2026-07-07": OutingEntry, "2026-07-12": OutingEntry }

type FavoritePlace = {
  id: string;
  name: string;
  category?: string;
  memo?: string;
  lat: number;
  lng: number;
};

// LocalStorage key: "routie:favorites"
```

`src/hooks/useOutings.ts`가 날짜별 일정 구조를, `src/hooks/useFavorites.ts`가 즐겨찾기 배열을 감쌉니다.

- `useOutingsMap()` — 날짜별 일정 전체 맵이 필요할 때 (달력 화면에서 일정 있는 날짜 표시 등)
- `useOuting(date)` — 특정 날짜의 `places`/`setPlaces`만 다룰 때 (오늘/지정 외출 화면). 오늘 날짜에 처음 진입하면 예시 데이터를 한 번 시딩합니다.

"오늘 외출"은 오늘 날짜 key를, "지정 외출"은 달력에서 선택한 날짜 key를 그대로 사용하므로 같은 화면 컴포넌트(`OutingScreen`)를 재사용할 수 있습니다.

## 지정 외출(달력) 기능

- `/calendar`에서 월 단위 달력을 보여주고, 일정이 있는 날짜는 점(●) 표시로 구분합니다.
- 날짜를 선택하면 하단에 해당 날짜의 일정 요약(또는 "예정된 외출이 없어요")이 표시됩니다.
- "+ 새 외출 만들기"(또는 일정이 있으면 "일정 보기") 버튼을 누르면 `/outing/[선택한 날짜]`로 이동합니다.
- 그 화면에서 장소를 추가하면 선택한 날짜의 일정으로 저장되며, 달력으로 돌아가 같은 날짜를 다시 선택하면 저장된 일정이 그대로 보입니다.
- **연도 바로 이동**: 달력 상단의 "OOOO년 O월" 텍스트를 누르면 연도 선택 Bottom Sheet가 뜹니다(`Calendar.tsx`). 월을 여러 번 넘기지 않고도 먼 과거/미래 연도로 바로 이동할 수 있게 하기 위함입니다.
  - 목록 범위는 **현재 보고 있는 연도와 오늘 연도 중 더 먼 쪽을 기준으로 앞뒤 20년씩**(`YEAR_PICKER_PADDING`) — 이미 몇 년 전/후를 보고 있는 상태에서 다시 열어도 그 연도가 항상 목록 안에 있습니다.
  - **오늘이 속한 연도**는 진한 파란색(`bg-primary`) 채움으로, **현재 보고 있는(오늘이 아닌) 연도**는 연한 파란색(`bg-accent`) 채움으로 구분됩니다 — day 그리드에서 "오늘"과 "선택한 날짜"를 구분하는 것과 같은 우선순위·색 규칙입니다(아래 "## Routie 메인 컬러 통일" 참고).
  - 시트가 열리면 현재 보고 있는 연도가 화면 중앙에 오도록 자동 스크롤됩니다(`scrollIntoView`).
  - 연도를 고르면 **보고 있던 월은 그대로 두고 연도만 바뀐 채** 즉시 달력이 갱신되고 시트가 닫힙니다(`calendar/page.tsx`의 `handleSelectYear`는 `viewYear`만 바꿉니다). 이전/다음 달 이동과 날짜 선택 흐름은 그대로입니다.

## 즐겨찾기

(홈 화면 메뉴명/헤더 제목 모두 "즐겨찾기"입니다 — 예전에는 "즐겨찾기 장소"였는데, "장소"를 빼서 더 짧고 간결하게 바꿨습니다. `routie:favorites`에 저장된 데이터 자체(예: `FavoritePlace`)나 훅 이름(`useFavorites`) 등 코드 상의 개념은 그대로 "즐겨찾기 장소"를 쓰지만, 화면에 보이는 문구만 바뀐 것입니다.)

- `/favorites`는 `routie:favorites`에 저장된 실제 데이터를 보여줍니다. 최초 진입 시 예시 데이터(온실카페·리움미술관·한강뷰 레스토랑)로 한 번 시딩되고, 이후에는 사용자가 저장/삭제한 내용이 그대로 유지됩니다.
- **카테고리 카드 → 항목별 아코디언, 2단 구조**(`app/favorites/page.tsx`): 이제 평면 목록이 아니라 `SavedPlacesPicker.tsx`와 같은 방식으로 먼저 **카테고리별 카드**(예: "카페 1개")로 묶여서 보이고, 카드를 누르면 그 안의 즐겨찾기들이 펼쳐집니다(`groupFavoritesByCategory`/`useCategories` 재사용, `expandedCategories: Set<string>`로 **여러 카테고리를 동시에** 펼칠 수 있습니다). 펼쳐진 카테고리 카드 안에서 항목 하나를 누르면 그 항목만 다시 펼쳐지며 메모(있으면, 없으면 "메모 없음")가 나타납니다(`expandedId` state, 한 번에 하나만 펼쳐지는 기존 동작 그대로 유지).
  - 화살표 아이콘은 카테고리 카드는 `ChevronDown`(펼치면 180도 회전), 그 안 항목은 기존과 동일하게 `ChevronRight`(펼치면 90도 회전)를 씁니다 — 두 계층이 서로 다른 회전 방향이라 "지금 내가 어느 레벨을 펼치고 있는지" 시각적으로 구분됩니다.
  - 펼침/접힘은 두 계층 모두 `framer-motion`의 `height: 0 → "auto"` 애니메이션으로 부드럽게 처리됩니다(`AnimatePresence` + `motion.div`, 200ms `easeOut`).
  - **카테고리가 없는 예전 데이터도 안전합니다**: `favorite.category`가 없거나 빈 값이면 항상 "기타"로 취급되어 그 카테고리 카드 안에 묶입니다 — 카테고리 기능이 생기기 전에 저장된 즐겨찾기라도 깨지지 않고 그대로 보입니다.
- **접힌 상태에서도 바로 되는 빠른 액션 3가지** — 각 항목 행에 아이콘 3개가 있고, 펼침용 화살표+이름 버튼과는 별개의 형제 버튼이라 눌러도 아코디언이 열리지 않습니다. 오늘 추가/날짜 선택/삭제가 펼치기 전에도 바로 되므로, 펼친 화면에는 이 버튼들을 중복해서 두지 않고 메모만 보여줍니다(예전엔 펼친 화면에도 같은 버튼 3개가 또 있어서 중복이었습니다).
  - **연필(이름 수정)** → 화살표+이름 버튼 바로 오른쪽에 붙어 있습니다(이름이 길어 잘리더라도 연필은 항상 이름 옆에 옵니다). 누르면 이름 수정 Bottom Sheet가 뜨고, 저장하면 그 즐겨찾기의 `name`만 바뀝니다(`openRename`/`handleRenameSubmit`).
  - **+(일정에 추가)** → 카드 오른쪽 끝(연필과는 `flex-1` 스페이서로 분리) 두 아이콘 중 왼쪽. "오늘 외출에 추가"/"날짜 선택 후 추가" 두 옵션이 있는 작은 Bottom Sheet가 뜹니다(`handleAddMenuToday`/`handleAddMenuDatePicker`).
  - **휴지통(삭제)** → 카드 오른쪽 끝, 확인 없이 바로 즐겨찾기 목록에서 제거됩니다.
- 추가할 때마다 하단에 확인 토스트가 잠깐 표시됩니다.
- **즐겨찾기 이름 수정이 지원됩니다**(위 연필 아이콘). 카테고리/좌표/메모 수정은 여전히 지원하지 않습니다 — 필요하면 삭제 후 다시 저장하세요.
- **"저장한 장소 불러오기"**(`SavedPlacesPicker.tsx`, `groupFavoritesByCategory`/`src/lib/favoriteGroups.ts`, 오늘/지정 외출 화면의 "+" → 장소 추가에서 접근)도 `/favorites`와 같은 카테고리 카드 구조를 씁니다 — 카테고리 헤더(예: "카페 3개")를 펼치면 그 안의 장소들이 나오고, 여러 카테고리를 동시에 펼칠 수 있습니다(처음엔 모두 접힌 채로 시작). **다만 장소를 고른 뒤의 동작이 다릅니다**: 여기서 장소를 탭하면 바로 일정에 추가되는 게 아니라, "장소 추가" 화면으로 돌아가 그 값(이름/좌표/메모)으로 폼이 미리 채워집니다 — 사용자가 내용을 확인/수정한 뒤 "추가하기"를 눌러야 실제로 일정에 반영됩니다(아래 "장소 추가/수정" 절의 "저장한 장소 불러오기" 흐름 참고). 이전에는 탭하는 즉시 확인 없이 바로 추가됐는데, 검토·수정 기회 없이 일정에 꽂히는 문제가 있어 이렇게 바뀌었습니다.

## Routie 메인 컬러 통일 (#4F7FFF)

앱 전반의 "선택됨/강조" 표시를 메인 컬러(`--primary`, `#4F7FFF`)와 그 연한 톤(`--accent`, 옅은 파란색) 두 가지로 통일했습니다. 둘 다 이미 `globals.css`에 정의돼 있던 디자인 토큰(`bg-primary`/`text-primary-foreground`, `bg-accent`/`text-accent-foreground`)이라 새 색을 추가하지 않고 그대로 재사용했습니다 — 어차피 `--accent`(연한 파란)/`--accent-foreground`(진한 파란)가 요청받은 `#EAF2FF`/`#2563EB`와 거의 같은 톤이라, 새 하드코딩 색을 쓰는 것보다 기존 토큰을 재사용하는 쪽이 "통일감"이라는 목표에 더 맞습니다.

- **방문 시간 선택 UI 교체** (`TimePicker.tsx`): 기존엔 브라우저 네이티브 `<input type="time">`을 썼는데, 네이티브 위젯은 OS/브라우저가 직접 그려서 오전/오후 순서나 선택 항목 색상을 우리 CSS로 바꿀 수 없었습니다. 이제 Bottom Sheet 안에 **오전(00~11시) → 오후(12~23시)** 순서로 정각 목록을 직접 그리고, 선택된 시간은 `bg-accent`/`text-accent-foreground`(연한 파란 배경 + 진한 파란 글자)로 표시하며 `rounded-lg`로 살짝 둥근 모서리를 줬습니다. 맨 위의 "시간 선택 안 함"으로 값을 비울 수 있습니다(시간은 선택 항목이라 필수가 아닙니다). `PlaceForm.tsx`의 "방문 시간" 입력이 이 컴포넌트로 교체됐습니다.
- **캘린더 색상 개편** (`Calendar.tsx`): 날짜 그리드와 연도 선택 목록 모두 같은 우선순위를 씁니다.
  - **오늘(또는 오늘 연도)**: `bg-primary` 진한 파란 배경 + 흰 글자로 가장 강조됩니다.
  - **오늘이 아니면서 선택된 날짜(또는 지금 보고 있는 연도)**: `bg-accent` 연한 파란 배경 + `text-accent-foreground` 진한 파란 글자.
  - 달력을 열면 기본 선택값이 "오늘"이라 두 조건이 자주 동시에 참이 되는데, 그럴 땐 **오늘 스타일이 우선**합니다(진한 채움). 그래서 다른 날짜를 고르면 오늘은 계속 진하게, 방금 고른 날짜는 연하게 — 두 정보(오늘이 언제인지, 지금 뭘 보고 있는지)가 동시에 눈에 들어옵니다.
  - 일정이 있는 날짜의 작은 점 표시(`isMarked`)는 원래도 `bg-primary`라 그대로 유지했습니다.
  - 기본(선택되지 않은) 날짜/연도의 hover 배경을 회색(`hover:bg-muted`)에서 **연한 파란색(`hover:bg-accent`)**으로 바꿨고, 이전/다음 달 화살표 아이콘도 회색(`text-foreground`)에서 **파란색(`text-primary`)**으로 바꿨습니다.

## PWA 서비스워커와 로컬 개발 시 흰 화면 문제

`PwaRegister.tsx`는 **프로덕션 빌드(`NODE_ENV === "production"`)에서만** 서비스워커(`public/sw.js`)를 등록합니다. 개발 모드(`npm run dev`)에서는 등록하지 않는 것을 넘어, **이미 등록되어 브라우저에 남아있는 서비스워커/캐시가 있다면 매번 자동으로 정리(unregister + 캐시 삭제)**합니다.

- **왜 필요했나**: `sw.js`는 모든 GET 요청을 "네트워크 우선, 실패 시 캐시 폴백"으로 가로채 캐싱합니다. 개발 중에는 코드를 저장할 때마다 새로운 JS 청크 파일이 생성되는데, 브라우저에 이미 등록된 서비스워커가 이전 세션의 오래된 청크를 계속 캐시로 들고 있으면 새로고침해도 화면이 예전 상태로 멈춰 있거나, 이미 존재하지 않는 청크를 요청하다 실패해서 **페이지가 하얗게 뜨는 문제**로 이어지기 쉽습니다. 이 문제는 코드 자체의 오류가 아니라서 `npm run build`나 서버 응답 확인만으로는 재현/발견되지 않고, 실제로 그 브라우저에서 이전에 한 번이라도 사이트를 열어 서비스워커가 등록된 적이 있어야만 나타납니다.
- **자동 정리 로직** (`PwaRegister.tsx`): 개발 모드에서 마운트되면 `navigator.serviceWorker.getRegistrations()`로 현재 origin에 등록된 서비스워커를 모두 찾아 `unregister()`하고, `caches.keys()`로 찾은 모든 Cache Storage 항목을 `caches.delete()`로 지웁니다. 그래서 예전에(이 가드가 생기기 전에, 또는 로컬에서 프로덕션 빌드를 열어봐서) 서비스워커가 남아있었더라도, 개발 서버를 새로고침하는 순간 자동으로 정리되고 그다음부터는 서비스워커 없이 항상 최신 코드로만 뜹니다 — 개발자 도구를 열어 수동으로 지울 필요가 없습니다.
- **그래도 화면이 하얗게 남아있다면** (자동 정리 로직 자체가 아직 실행되지 못한, 즉 이미 캐시된 오래된 페이지가 떠서 새 JS를 아예 못 받아온 극단적인 경우) 아래를 수동으로 진행하세요.
  1. 개발자 도구(F12) → Application(또는 Storage) 탭 → Service Workers → 목록에서 **Unregister**
  2. 같은 탭의 Cache Storage에서 `routie-cache-v1` 캐시가 있으면 우클릭 → **Delete**
  3. 하드 새로고침(Ctrl+Shift+R / Cmd+Shift+R) — 그래도 안 되면 시크릿/프라이빗 창에서 다시 열어 확인
- 프로덕션 배포본(Vercel 등)에서는 계속 정상적으로 서비스워커가 등록되어 오프라인/PWA 설치 기능이 그대로 동작합니다 — 이번 변경은 로컬 개발 환경에만 영향을 줍니다.

1. GitHub에 저장소를 push합니다.
2. [Vercel](https://vercel.com)에서 New Project로 저장소를 Import합니다.
3. Environment Variables에 `NEXT_PUBLIC_NAVER_MAP_CLIENT_ID`를 등록합니다 (없어도 Mock Map으로 배포는 정상 동작).
4. 네이버클라우드플랫폼 Application 설정에 배포 도메인을 서비스 URL로 추가합니다.
5. Deploy 실행 시 자동으로 빌드/배포됩니다.

## 제외 범위 (MVP)

- 로그인/회원가입, 서버·데이터베이스 연동 (LocalStorage만 사용)
- 실제 길찾기 기반 경로 계산, 최적 경로 추천 (거리/도보시간은 직선거리 추정치)
- 즐겨찾기 수정 (삭제/추가만 지원)
- 일정 공유, 다크 모드
